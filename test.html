Test & Reflect

How we validated the system, what we discovered, and what we'd do differently.

Testing Approach

We tested Keep Attendance across multiple scenarios to evaluate accuracy, reliability, and ease of use.

Testing was conducted iteratively throughout the development process. We began with controlled single-person scenarios to establish a baseline for detection accuracy, then moved to more realistic multi-person and variable-lighting conditions.

Each test round was used to tune the model's confidence threshold, improve detection stability, and refine the web dashboard's real-time update behavior. Our goal was to ensure the system could operate reliably in a standard indoor environment without any special setup.

We also evaluated the deployment process itself — how quickly a new instance could be set up, how well the system handled network interruptions, and how intuitive the interface was for a first-time user.

What Worked

✓

Reliable detection in standard conditions

Under consistent indoor lighting, the system accurately detected and logged presence with strong reliability.

✓

Real-time dashboard updates

Attendance records appeared in the web interface with minimal delay, making real-time monitoring practical.

✓

Fast and simple deployment

A fresh instance of the system could be set up and operational in minutes, validating the portability goal.

✓

Browser-based accessibility

The dashboard worked correctly across multiple devices and browsers without any installation required.

Challenges We Faced

01

Variable lighting conditions

Detection accuracy dropped in low-light or high-contrast environments. Future iterations would benefit from preprocessing steps to normalize frame exposure before analysis.

02

Confidence threshold tuning

Finding the right balance between sensitivity and specificity required significant iteration. Too low a threshold generated false positives; too high missed valid detections.

03

Network latency on slower connections

On networks with higher latency, the dashboard refresh rate lagged behind actual detections. Optimizing the event-streaming mechanism would improve this.

04

Differentiating individuals

The current system detects presence rather than identity. Tracking specific individuals would require facial recognition, which introduces additional complexity and privacy considerations.

Reflections & Next Steps

What we learned and where we'd take this project next.

Improved lighting robustness

Implementing adaptive exposure normalization would make the system reliable in a wider range of physical environments.

Individual identification

Adding an opt-in facial recognition layer — with appropriate privacy controls — would enable named attendance records rather than presence-only detection.

Data export & integrations

Exporting attendance data to CSV, Google Sheets, or LMS platforms would dramatically increase the system's practical value.

Multi-camera support

Scaling to multiple camera feeds simultaneously would allow coverage of larger spaces and more complex environments.
